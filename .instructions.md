# ⚠️ 삼미랙 프로젝트 총괄 작업 원칙 (필독, 모든 레이어 공통)

**이 문서는 삼미랙 프로젝트 내 API, 데이터베이스, 프론트엔드, 백엔드, 서비스 설계 등 모든 레이어에 적용되는 공통 작업 원칙입니다.**

- 프로젝트 전체에서 동일하게 따라야 하는 표준 워크플로우
- 특정 파일/모듈의 편향적 “하드코딩 지침”이 아님
- 모든 Agent는 작업 시작 전 반드시 이 문서를 읽고 확인 체크를 수행

---

## 1️⃣ 절대 규칙: 전체 의존성(코드 + 데이터 + API) 먼저 파악

### ❌ 나쁜 케이스 (절대 금지)
```
user: "aa를 위해 dd라는 게 필요하다"
  ↓
Agent: 현재 상황 파악 안 함 → 일반적 상황 추론 → 부분 수정
  ↓
결과: 의존성 불일치 → 오류
```

### ✅ 올바른 케이스 (필수)
```
user: "aa를 위해 dd 구현"
  ↓
Agent: "aa에 영향 받는 모든 코드(1,2,3)와 엮인 코드(z,d,c) 먼저 전체 읽기"
  ↓
"이 부분들이 연결되어 있으므로 다음을 모두 수정해야 함" 명시
  ↓
모든 엮인 코드를 고려해서 맥락정보를 유지하며 수정 (불완전한 수정 절대 금지)
```

## 2️⃣ 필수 확인 리스트 (모든 작업 전)

### 단계 1: 전체 의존성 지도 그리기
- [ ] 수정할 함수/파일의 **호출처**(caller) 전부 찾기
- [ ] 수정할 함수가 사용하는 **의존성**(dependency) 전부 찾기
- [ ] 반건 return 값이 **다른 곳에서 어떻게 사용**되는지 확인
- [ ] 상위/하위 함수 체인 문서화

### 단계 2: 실 데이터 기반 검증 (모든 도메인 공통)
- [ ] 실제 운영/테스트 데이터를 사용해 E2E 흐름 검증 (특정 발주서에 국한되지 않음)
- [ ] 검증 범위는 항상 모든 레이어에 걸쳐야 함
  - API 스팩, DB 스키마, 백엔드 로직, 프론트엔드 상태, 배치/크론 작업 등 포함
- [ ] 핵심 플로우 및 필드(예: ssSource, status, id, quantity 등)가 처음부터 끝까지 전달되는지 확인

### 단계 3: 핵심 필드 일관성 & 확장성 체크
```python
# ❌ 나쁜 예
res.append({
  "name": "기둥",
  "rackType": rack_type,
  # ← ssSource가 없다! → 나중에 정렬할 때 터짐
})

# ✅ 좋은 예
res.append({
  "name": "기둥",
  "rackType": rack_type,
  "ssSource": "main",  # ← 반드시 있어야 함
})
```

## 3️⃣ 스마트스토어 주문 처리 플로우 (순서 중요)

```
1. order_listener.build_grouped_document()
   ├─ mains 주문 → generate_bom_for_rack() → BOM 자재
   │  └─ 각 res.append는 반드시 "ssSource": "main" 포함
   ├─ addons 주문 → build_material_item() → addon 자재
   │  └─ 반드시 "ssSource": "addon" 포함 (이미 있음)
   │
2. materials 중복 제거 & 병합
   └─ key에 "ssSource" 포함시켜야 메인/애드온 구분 유지
   │
3. sorted(materials, key=_material_sort_key)
   ├─ rackType 기준
   ├─ ssSource_priority 기준 (메인=0, 애드온=1)
   └─ group_priority 기준 (materialOrder.json)
   │
4. 최종 정렬 결과
   └─ 메인 부품들 (기둥, 로드빔, 선반, ...)
   └─ 애드온 부품들 (메트그레이선반, ...)
```

---

## 4️⃣ 최근 이력 및 참고 사항 (2026-04-03)

### 배경
- 사례: 메인/애드온 자재 정렬 문제 발견 후 수정 적용
- 이력 목적: 추후 동일 이슈 재발 방지 및 원인 파악 기록

### 이력 요약
- `generate_bom_for_rack()`에 `"ssSource": "main"` 추가 적용
- `build_material_item()`는 `"ssSource": "addon"` 적용 유지
- `_material_sort_key()`는 ssSource를 2순위로 정렬 (main 0, addon 1)

### 체크리스트 (중요: 이건 범용 검증 절차어야 함)
1. [ ] materialOrder.json은 **절대 수정 금지** (칸 일관성 옵션)
2. [ ] 주요 파이프라인 필드(예: ssSource, status, id, quantity, rackType) 엔드투엔드 전달 확인
3. [ ] `merged_mats` 병합 키에 ssSource 포함 여부 확인
4. [ ] `_material_sort_key`가 메인/애드온 우선순위 분리 수행 확인
5. [ ] 특정 케이스(예: SS-63411913 등)만 아니라 “여러 사례”로 검증

## 5️⃣ Agent 체크리스트 (이 문서를 읽은 후)

- [ ] 새 필드/변경사항이 **전체 파이프라인 통과**하는가?

### 작업 후
- [ ] **검증 코드**를 실제로 돌렸는가?
